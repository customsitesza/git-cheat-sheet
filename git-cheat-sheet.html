Every command you need to manage your code, from first commit to branching and beyond.

Setting Up a New Repo

Before you can push code to GitHub, you need to create an empty repo on GitHub and then link your local project folder to it. Don't add a README or .gitignore on GitHub — keep it completely empty.

Step-by-step from scratch

Open your terminal in the project folder and run these commands in order:

1
git init
Turns your folder into a Git repository. Creates a hidden .git folder that tracks everything.
2
git remote add origin <url>
Links your local repo to the GitHub repo. origin is the nickname for the remote. The URL is copied from your GitHub repo page.
3
git add .
Stages all your files. The dot means "everything in this folder." This tells Git which files to include in the next commit.
4
git commit -m "Initial commit"
Saves a snapshot of your staged files with a message describing the change.
5
git branch -M main
Renames your default branch to main, which is GitHub's standard.
6
git push -u origin main
Uploads your commit to GitHub. The -u flag links your local branch to the remote so future pushes only need git push.

.gitignore

Create this file in your project root before your first commit to keep build files and secrets out of your repo:

__pycache__/
*.pyc
dist/
build/
*.spec
*.exe
.env
⚡

Everyday Workflow

Every time you make changes to your code, it's just three commands. Write clear commit messages — future you will thank present you.

# Stage all changes
git add .

# Commit with a descriptive message
git commit -m "Added lead export to CSV feature"

# Push to GitHub
git push
🌿

Branches

Branches let you work on features in isolation. Your main branch stays stable while you build and test on separate branches. When a feature is done, merge it back in.

Create & switch to a new branch

The -b flag creates and switches in one command.

git checkout -b feature/your-feature-name

Push a branch to GitHub (first time)

After this, plain git push works for that branch.

git push -u origin feature/your-feature-name

Merge a branch into main

Switch to main first, then pull in the feature branch.

# Switch to main
git checkout main

# Merge the feature
git merge feature/your-feature-name

# Push the updated main
git push

Delete a branch

Local and remote are separate — delete both when you're done.

# Delete locally
git branch -d feature/your-feature-name

# Delete from GitHub
git push origin --delete feature/your-feature-name

Naming convention

Prefix branches by type so you can tell at a glance what they're for:

feature/lead-export feature/sop-templates fix/converter-crash fix/dashboard-layout experiment/new-ui
↩️

Undoing Things

Made a mistake? Git has several ways to roll back. Pick the right one depending on how far back you need to go.

View commit history

See all your commits with their short hashes.

git log --oneline

Undo last commit (keep files)

Good for "I committed too early." Your code stays as-is but the commit is removed.

git reset --soft HEAD~1

Revert a specific commit (safe)

Creates a new commit that undoes the changes. Doesn't erase history.

git revert <commit-hash>

Reset to a specific commit (destructive)

Wipes everything after that commit. Use with caution — there's no undo.

git reset --hard <commit-hash>
git push --force
📦

Stashing

Need to switch branches but have uncommitted changes? Stash them temporarily and come back later.

# Save changes temporarily
git stash

# Switch branches, do your thing, then come back
git checkout feature/your-branch

# Restore stashed changes
git stash pop

Golden Rules

If it works, commit it You always want a known good version to revert to when things break.
Main = always stable Build features on branches, merge when they're tested and done.
Write clear commit messages Future you will thank present you when you need to find where a bug was introduced.
Build your .exe from main That way you always know it's the stable, complete version of your app.
